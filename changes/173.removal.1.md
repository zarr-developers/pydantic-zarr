Adopting `zarr-metadata` changed the field *types* of the loose `ArraySpec`/`GroupSpec` (the
field *names* are unchanged). Two user-visible behavioral differences:

- Sequence-valued fields are now tuples on the model and in `model_dump()` (Python mode): v2
  `ArraySpec.filters` and v3 `chunk_grid` `chunk_shape` are tuples rather than lists. This makes the
  parsed spec immutable, so a spec cannot be mutated in place after construction. (`to_json()`
  serialization is unchanged — both lists and tuples serialize to JSON arrays.) Code that read
  `spec.filters` expecting a mutable `list` (e.g. `.append(...)` or `isinstance(x, list)`) must
  adjust.
- The loose v3 named-config fields (`chunk_grid`, `chunk_key_encoding`, `codecs`) are now typed
  with `zarr-metadata`'s generic metadata type (`str | NamedConfigV3`), and `fill_value` (v2/v3)
  and `dtype`/`data_type` with `zarr_metadata.JSONValue` / the zarr-metadata dtype types. As
  before, the loose specs validate these as *syntax only* — they accept any well-formed
  name-or-`{name, configuration}` value and do not check whether a given name is spec-valid or
  whether its configuration is well-formed. Use the `Core`/`Extra` strict families for that
  (dtype-aware `fill_value` checks, codec/grid internal validation, and spec-conformant rejection
  of names whose configuration is required).

This is a behavioural change for code that relied on the previous list-typed (now tuple)
attributes.
